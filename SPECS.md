1\. Общее описание проекта и Архитектура

Стек: React (18+), TypeScript, Tailwind CSS v4, сборщик Vite.



Хранение: localStorage.



Экспорт/Импорт данных: Осуществляется в формате JSON. Пользователь может выбрать временной промежуток для экспорта. Импорт может быть частичным (например, загрузить данные только за один день). Миграции схемы данных между версиями приложения не поддерживаются.



Библиотека продуктов: Все введенные вручную блюда и продукты, полученные по штрихкоду, автоматически сохраняются в единую локальную библиотеку для быстрого повторного использования. Кеширование запросов API происходит туда же.



2\. Обновленные модели данных (TypeScript)

Из-за требования "изменение продукта пересчитывает все связанные записи", мы разделяем сущности на Элемент библиотеки (базовые характеристики) и Запись в дневнике (ссылка на библиотеку + вес порции).



TypeScript

interface KBZHU {

&#x20; calories: number;

&#x20; proteins: number;

&#x20; fats: number;

&#x20; carbs: number;

}



// ---------------- БИБЛИОТЕКА ПРОДУКТОВ ----------------



// Базовый продукт в библиотеке (единый источник истины)

interface LibraryItem {

&#x20; id: string; // Уникальный ID продукта

&#x20; name: string;

&#x20; barcode?: string;

&#x20; baseKBZHU: KBZHU; // Всегда приведено к значениям на 100г для унификации

}



// ---------------- ДНЕВНИК ПИТАНИЯ ----------------



// Элемент списка (Съеденное блюдо)

interface FoodLogItem {

&#x20; id: string; // Уникальный ID конкретной записи в дневнике

&#x20; type: 'food';

&#x20; libraryItemId: string; // Ссылка на базовый продукт из библиотеки

&#x20; weight: number; // Вес съеденной порции в граммах

}



// Группа (Строго до 3 уровней вложенности)

interface FoodGroup {

&#x20; id: string;

&#x20; type: 'group';

&#x20; name: string;

&#x20; items: Array<FoodLogItem | FoodGroup>; 

&#x20; isExpanded?: boolean;

}



// Приём пищи

interface Meal {

&#x20; id: string;

&#x20; name: string; 

&#x20; items: Array<FoodLogItem | FoodGroup>;

}



// Запись одного дня

interface DayRecord {

&#x20; date: string; // 'YYYY-MM-DD'

&#x20; meals: Meal\[];

}



// ---------------- НАСТРОЙКИ И КОРЕНЬ ----------------



interface UserSettings {

&#x20; targetKBZHU: KBZHU;

&#x20; defaultMeals: string\[]; // По умолчанию: \['Завтрак', 'Перекус', 'Обед', 'Перекус', 'Ужин']

}



// Корневое состояние хранилища (AppState)

interface AppState {

&#x20; library: Record<string, LibraryItem>; // Словарь всех продуктов: ключ - id

&#x20; records: Record<string, DayRecord>;   // Словарь дней: ключ - дата

&#x20; settings: UserSettings;

}

3\. Спецификация функционала

3.1. Навигация и Дни

Отображение дат: В верхней части экрана располагаются кружочки, обозначающие дни текущей недели.



Доступность дней: Пользователь может редактировать любую историю (прошлые недели/дни). Вперед можно заходить только в рамках текущей недели (планирование на ближайшие дни). Будущие недели недоступны для выбора.



Переход суток: При наступлении 0:00 интерфейс автоматически переключается на новый день.



Инициализация дня: Если выбранный день пуст, он генерируется на основе массива settings.defaultMeals.



Копирование: Реализована функция копирования блюда или целого приема пищи на другой день (функционал "Вчера ел то же самое").



3.2. Библиотека, Блюда и API Штрихкодов

Добавление через API: При сканировании/вводе штрихкода продукт запрашивается из API. Если найден — он автоматически сохраняется в library (кешируется). Пользователю остается ввести только вес съеденной порции. Редактирование данных от API перед сохранением не требуется.



Ручное добавление: Пользователь вводит название и КБЖУ. Это создает новую запись в library.



Редактирование продукта: Если пользователь открывает библиотеку и изменяет КБЖУ продукта, эти изменения применяются к объекту LibraryItem. Приложение должно пересчитать статистику для всех дней, где этот продукт был использован (поскольку лог ссылается на libraryItemId, при рендеринге история автоматически отобразит новые значения). Историю старых значений (до редактирования) хранить не нужно.



3.3. Группы и Приёмы пищи

Максимальная вложенность: Строго 3 уровня (Группа -> Группа -> Группа -> Блюдо). UI и логика Drag-n-Drop должны блокировать попытку перетащить элемент глубже этого лимита.



Настройки приемов пищи: Пользователь может изменить стандартное количество и названия приемов пищи в settings.defaultMeals (например, оставить только 3). Изменения применяются ко всем новым дням.



3.4. Статистика и Графики

Агрегация (Неделя / Месяц): По умолчанию показывается среднее значение за выбранный период. В UI есть переключатель для отображения общей суммы (Total).



Нулевые дни: Если в выбранном периоде есть дни без записей (0 ккал), они учитываются при расчете среднего значения, тем самым опуская среднюю цифру (стимул не пропускать заполнение).



График БЖУ за день: Pie Chart (круговая диаграмма). Показывает соотношение калорий (энергии), полученных из Б, Ж и У (1г Б/У = 4 ккал, 1г Ж = 9 ккал), а не вес в граммах.



Цели: Прогресс-бар для белков и остальных макронутриентов на основе settings.targetKBZHU.



4\. Обработка граничных случаев (Edge Cases)

Удаление продукта из библиотеки, который есть в истории:



Решение: Запретить "жесткое" удаление продукта из library, если на него есть ссылки в records. Вместо этого добавить продукту флаг isArchived: true, чтобы скрыть его из поиска, но сохранить возможность отрисовки прошлых дней.



Зацикливание и лимит вложенности (Drag-n-Drop):



Решение: В обработчике onDragOver (или аналоге из dnd-kit) вычислять текущую глубину узла + глубину перетаскиваемого дерева. Если сумма > 3, отменять возможность drop.



Недоступность API штрихкодов / CORS:



Решение: Показать уведомление об ошибке и перевести пользователя на экран ручного создания продукта в библиотеку.



Смена настроек приемов пищи (backward compatibility):



Решение: Если пользователь меняет defaultMeals в настройках, уже заполненные прошлые дни не изменяются. Изменения применяются только при инициализации пустых дней.



Частичный импорт данных:



Сценарий: Импорт одного дня, который содержит продукты, отсутствующие в текущей library.



Решение: Файл экспорта должен включать не только DayRecord, но и фрагмент library со всеми продуктами, которые использовались в экспортируемом периоде, чтобы при импорте восстановить зависимости.






Дополнения к спецификации функционала и интерфейса

1\. Интерфейс: Обработка коллизий одинаковых названий (Name Disambiguation)

Логика работы: При отображении блюд как в самом Дневнике питания, так и в поисковой выдаче библиотеки продуктов, приложение должно проверять наличие дубликатов имен. Если в системе (library) существуют два или более продуктов с абсолютно идентичным name (без учета регистра), рядом с их названием должен рендериться уникальный хэш-идентификатор.



Визуальное отображение:



Идентификатор берется из id продукта (например, первые 4–6 символов UUID или короткого хеша для читаемости).



Формат: Название блюда #a1b2



Если название уникально, серый ID не отображается, чтобы не перегружать интерфейс.



2\. Интерфейс: Компонент выбора блюда из библиотеки (Picker / Dropdown)

Отображение подсказок: В списке результатов поиска (выпадающем списке или модальном окне библиотеки) каждый элемент должен иметь четкую двухстрочную структуру:



Строка 1 (Основная): Название продукта (плюс серый #id, если есть совпадение имен).



Строка 2 (Вспомогательная): Плотность калорий, расположенная строго под названием.



Формат: ХХХ ккал / 100 гр, выполненный уменьшенным и более блеклым шрифтом (text-xs text-slate-500 в Tailwind v4).

Дополнение к граничным случаям (Edge Cases)

Генерация ID при импорте: При частичном импорте данных за определенный день из JSON-файла может возникнуть ситуация, когда импортируемый продукт имеет то же имя, что и существующий в базе, но другие КБЖУ.



Решение: Поскольку связь идет через ID, приложение запишет его как новый продукт в library со своим уникальным ID. Благодаря механизму дедупликации имен, в интерфейсе они автоматически разделятся с помощью серых тегов #hash, не ломая логику и не требуя от пользователя переименования вручную.
